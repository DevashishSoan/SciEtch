Vision Statement:  SciEtch is the fastest, most constrained design tool built exclusively for academic researchers — turning a raw abstract into a publication-ready graphical schematic in under 60 seconds.

1. Executive Summary
SciEtch solves a persistent pain point in academic publishing: researchers produce ground-breaking science but lack the time or design skills to create the high-quality graphical abstracts increasingly required by journals (Cell, Nature, Science, ACS), preprint servers (arXiv, bioRxiv), and open-access repositories (Zenodo, OSF).

Existing tools — Adobe Illustrator, Canva, BioRender — are either too broad, too expensive, or locked behind institutional licenses. SciEtch is ruthlessly focused: paste an abstract, let the AI extract the key methodology, customize a generated schematic, and export a print-resolution PDF — all in one workflow.

Market Opportunity:  Over 3 million peer-reviewed papers are published per year. More than 70% of top-tier journals now mandate or strongly encourage graphical abstracts. The TAM for design tooling in academic publishing exceeds $1.2B globally.

2. Problem Statement
2.1 Core Pain Points
Researchers spend 3–8 hours on a single graphical abstract using general-purpose tools not designed for science.
BioRender is expensive ($35–$99/mo), biology-centric, and lacks NLP-driven layout generation.
Canva and PowerPoint produce graphical abstracts that look amateurish next to peer-reviewed publication standards.
arXiv, Zenodo, and journal submission portals have strict file-size and resolution requirements that most researchers don't know how to meet.
Collaborating PI and postdoc on figure iterations involves emailing .pptx files back and forth — no version control, no commenting.

2.2 User Frustration Snapshot
Researcher Quote (interviews):  "I spent an entire Sunday afternoon moving boxes around in Illustrator just to make one figure for the journal submission. The science took two years — the figure took longer than it should."

PI Quote:  "We reject Canva graphics immediately — they look unprofessional. But we don't have time to teach every grad student Illustrator."

5. User Stories & Acceptance Criteria
5.1 Abstract Ingestion & NLP Parsing
As a researcher, I want to paste my abstract and have the app identify the core methodology, inputs, outputs, and key steps, so that I don't have to manually create a flowchart from scratch.
AC: Parser extracts ≥3 distinct process nodes from any 150–300 word abstract within 8 seconds.
AC: Confidence score shown per extracted node; user can edit/reject any node before layout generation.
AC: Parser works on abstracts from biology, chemistry, physics, CS, and social sciences.

5.2 Layout & Schematic Generation
As a user, I want to see a clean, structured schematic generated automatically from my parsed nodes, so I have a professional starting point rather than a blank canvas.
AC: Layout renders within 3 seconds of parsing confirmation.
AC: Generated schematic adheres to one of five layout templates (linear flow, cyclic process, hierarchical, comparison, hybrid).
AC: All generated elements snap to an invisible grid; no overlapping boxes on initial render.

5.3 Drag-and-Drop Component Editor
As a researcher, I want to customize the generated schematic using a scientific component palette, so I can add domain-specific visuals without learning Illustrator.
AC: Component library contains ≥50 components across categories: flowchart, biology, chemistry, data/stats, physics.
AC: Components snap to grid with 8px precision.
AC: Undo/redo supports ≥30 steps.
AC: Text inside any component is editable with inline click.

5.4 Export
As a researcher, I want to download my schematic as a high-resolution PDF or PNG ready for journal submission, so I don't have to worry about DPI or file size limits.
AC: PDF export at 300 DPI minimum; file size ≤5MB for a standard abstract figure.
AC: PNG export at 300 DPI with transparent background option.
AC: SVG export for vector editing in downstream tools.
AC: Journal preset profiles (Nature, ACS, Elsevier) enforce correct dimensions and color modes automatically.

5.5 Collaboration
As a PI, I want to leave inline comments on a shared figure, so I can give feedback without emailing files back and forth.
AC: Share link generates in <2 seconds; no account required for comment-only access.
AC: Inline comments pin to individual components; resolved comments archived.
AC: Version history stores up to 20 snapshots per figure.

7.2 NLP Pipeline Detail
The text extraction pipeline runs as a stateless Python microservice exposed over REST:

Input: Raw abstract text (UTF-8, 50–600 words).
Step 1 — Sentence segmentation: spaCy en_core_web_sm pipeline.
Step 2 — Entity recognition: custom NER model fine-tuned on 12,000 annotated scientific abstracts across 6 disciplines. Labels: METHOD, VARIABLE, ORGANISM, INSTRUMENT, DATASET, OUTCOME.
Step 3 — Relation extraction: dependency parsing identifies subject-verb-object triples linking entities into process edges.
Step 4 — Layout scoring: a heuristic scorer maps extracted triples to one of five layout archetypes (linear, cyclic, hierarchical, comparison, hybrid) based on edge topology.
Step 5 — JSON output: nodes (id, label, type, confidence) + edges (source, target, label) returned to React frontend for rendering.

Performance Target:  End-to-end pipeline latency < 3 seconds (p95) for abstracts up to 400 words, measured at the FastAPI endpoint under normal load.

7.3 Canvas Editor Architecture
The editor is a React component tree with a Zustand store managing the full figure state (nodes, edges, styles, history stack).
Each component in the figure is a Konva.Group containing a shape layer and a text layer; all transforms are stored as normalized relative coordinates (0–1 space) and mapped to canvas pixels at render time to support responsive resizing.
Undo/redo is implemented as an immutable history array with a cursor pointer; each mutation pushes a new state snapshot. The stack is capped at 30 entries to bound memory.
Export is triggered by serializing the Konva stage to a data URI, then sending it to the backend export service (Puppeteer for PDF, Sharp pipeline for PNG/SVG).

8. Design Principles & UX Constraints
8.1 The Three Constraints (Product Philosophy)
Constraint 1 — No Blank Canvas:  Users always start from a generated layout. The empty canvas is never shown. This eliminates decision paralysis and keeps output consistent.

Constraint 2 — Science-Only Components:  The component library contains exclusively scientific primitives. No clip-art, no generic shapes. Every component is semantically labeled (e.g., 'Western Blot Frame', 'Decision Gate', 'Data Pipeline Arrow').

Constraint 3 — Journal-First Export:  Export dialogs default to journal profiles. Users select a destination (Nature, arXiv, etc.) and the app enforces correct dimensions, color space (RGB vs CMYK), and file size automatically.

8.2 UI Layout
Left Panel — Component Library: searchable grid of scientific components, grouped by discipline. Drag onto canvas to place.
Center — Canvas Editor: the live Konva canvas with snap-to-grid enabled by default. Toolbar floats at top: select, pan, zoom, add text, add arrow.
Right Panel — Properties: context-sensitive panel showing fill/stroke, font, label, and component-specific options for the selected element.
Top Bar — Export & Share: Journal preset picker, Export button, Share link generator, Version history dropdown.

8.3 Accessibility & Internationalisation
WCAG 2.1 AA compliance for all interactive UI elements outside the canvas.
Keyboard shortcuts for all common editor actions (documented in Help overlay).
RTL layout support for Arabic, Hebrew, Farsi (abstract text direction only; canvas is language-agnostic).
i18n strings externalized from day one; initial launch in English only.

14. Appendix
14.1 Glossary
Term
Definition
Graphical Abstract
A single-panel figure that visually summarizes the key findings of a research paper, increasingly required by journals.
NER
Named Entity Recognition — an NLP task that classifies text tokens into predefined categories (e.g., METHOD, OUTCOME).
DPI
Dots Per Inch — print resolution. Journals require ≥300 DPI; screen resolution is typically 72–96 DPI.
DXA
Document XML Attribute unit used in .docx; 1 inch = 1440 DXA.
ORCID
Open Researcher and Contributor ID — a persistent digital identifier for researchers, widely used in academic publishing.
Konva.js
A 2D HTML5 canvas library that provides a high-level API for shapes, events, and animations — used for the figure editor.
spaCy
An open-source Python NLP library used for tokenization, POS tagging, dependency parsing, and NER.
Layout Template
A predefined spatial arrangement of nodes and edges (e.g., linear flow, cyclic process) applied to the extracted NLP graph.
